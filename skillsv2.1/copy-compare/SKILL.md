# 落地页文案竞品对比分析（直接回应文案专家版）

## 概述

从"谁的文字更能卖货"的角度，对我方文案（版本A）与竞品文案（版本B）进行深度对比和量化打分。

## 使用方式

```
/copy-compare https://competitor-url.com
/copy-compare https://competitor-url.com https://competitor-url2.com
/copy-compare competitor-copy.txt C:\path\to\our-copy.html
```

## 输入来源

- **竞品文案（版本B）**：URL（用WebFetch抓取，如失败用浏览器工具）或本地文件路径 或 直接粘贴文案
- **我们的文案（版本A）**：指定文件路径，或使用当前对话中最近生成的文案，或工作目录下最近的 *_Landing_Page_Copy*.md / *_Landing_Page_Copy*.html 文件

---

## 角色设定

你现在是一位顶级的"直接回应文案专家（Direct Response Copywriter）"。你只评判文案本身的销售杀伤力——即文字、结构、叙事对读者心理的操控能力。价格策略、套餐组合、退款政策等商业运营决策不在评判范围内。

## 业务背景

我们是一个纯卖货的护肤及功能性产品电商网站（品类：丰唇、脱毛、紧致、瘦身、疼痛缓解等）。

## 目标受众画像

年龄段：45岁-50岁+的中老年男性女性为主。

核心特征：
- **极度务实**：只关心"这东西对我有什么用"、"能不能解决我的痛苦"
- **防备心极重**：怕被骗，怕没效果，极度依赖真实评价和权威背书
- **注意力稀缺**：没耐心看长篇大论，喜欢大白话、直接给结论
- **决策靠情绪+信任**：不是被逻辑说服，是被画面感和"跟我一样的人"打动

---

## 重要排除项

以下内容属于商业运营决策，**不纳入评分**，评判时请直接忽略：
- 定价策略（单价、套餐阶梯、折扣幅度、价格锚定）
- 退款/保证政策（保证时长、退款条件）
- 支付方式与物流承诺

## 执行流程

### Step 1：抓取与整理

**竞品URL抓取（按优先级依次尝试）：**
1. 先用 WebFetch 直接抓取URL
2. 如果返回403/超时/失败 → 使用 mcp__Claude_in_Chrome 浏览器工具（navigate + get_page_text / read_page）
3. 如果浏览器工具也不可用 → 提示用户手动粘贴文案内容或提供本地文件
4. **每个URL必须确认拿到实际文案内容后再进入对比环节，不能跳过**

**我方文案读取：**
1. 读取用户指定的文件（优先 .md 或 .txt 格式）
2. 确认版本A（我方）和版本B（竞品）内容完整

### Step 2：纯文案销售力打分（总分100分）

#### 维度1：黄金开头与注意力锁定（20分）
- 标题和前3屏内容是否瞬间抓住50岁+人群的眼球？
- 是否一针见血指出了目标人群最痛的痛点？
- 标题读完后，读者是否有强烈冲动继续往下滑？

#### 维度2：信任链构建（25分）
- 权威背书是否可信且与目标人群认知匹配？（医生推荐、机构认证、媒体引用等）
- 用户证言是否具备"粗糙的真实感"？（是否有年龄/职业/城市等颗粒度？措辞是否像真人说话而非营销团队代笔？）
- 整体信任体系能否瓦解50岁+人群的"怕被骗"心理？

#### 维度3：利益翻译与欲望激发（30分）——【核心文案分，权重最高】
- 产品卖点是否翻译成了目标人群秒懂的"大白话利益"？（不说"促进胶原蛋白合成"，而是说"让松弛的肚皮重新紧回去"）
- 是否创造了具体的"未来画面"让读者代入？（镜子场景、试衣服场景、被人夸的场景）
- 成分/机制的科学信息是否做了"降维翻译"，而非直接甩术语？
- 读完后，读者是否产生了"我想变成文案里描述的那个人"的欲望？

#### 维度4：叙事结构与阅读推进力（15分）
- 文案是否有清晰的情绪曲线（痛点→共鸣→希望→证据→想象→行动）？
- 段落之间是否有"钩子"让50岁+读者持续往下滑，而非中途流失？
- 信息密度是否适配目标人群的注意力和认知能力？（是否太冗长/太学术/太散乱？）

#### 维度5：行动号召与临门一脚（10分）
- CTA按钮文案是否清晰、有行动感、有情绪驱动力？
- 下单路径在文案层面是否显得极其简单？（"只需5秒"、"撕开贴上睡觉"等）
- 文案结尾是否给了读者一个"现在就行动"的情绪推力？

### Step 3：输出对比报告

#### 报告格式（严格按此结构输出）

```
# 直接回应文案专家对比报告

## 第一部分：打分表

| 维度 | 版本A（我方） | 版本B（竞品） | 差距 |
|------|-------------|-------------|------|
| 1. 黄金开头与注意力锁定（20分） | X | X | +/-X |
| 2. 信任链构建（25分） | X | X | +/-X |
| 3. 利益翻译与欲望激发（30分） | X | X | +/-X |
| 4. 叙事结构与阅读推进力（15分） | X | X | +/-X |
| 5. 行动号召与临门一脚（10分） | X | X | +/-X |
| **总分（100分）** | **X** | **X** | **+/-X** |

## 第二部分：关键PK对比

逐维度直接对比，每个维度用2-3句话说清楚谁赢、为什么赢。
要求：
- 打分要严苛，不允许礼貌性给高分
- 用具体文案片段做对比，不做模糊评价
- 站在50岁+目标人群的视角评判

## 第三部分：最终判定

明确指出从纯文案销售力角度，版本X更能卖货。用一句话总结其致命优势。

## 第四部分：我方版本优化建议

无论最终判定结果如何，优化建议只针对版本A（我方文案）。
从以下两个角度各给出1条具体修改建议：

### 补短板
版本A在哪个维度失分最多？给出一条具体的文案修改方案：
- 精确到改哪段
- 原文是什么
- 改成什么
- 为什么这样改能提分

### 抄优点
版本B的文案中有哪一个具体写法/技巧值得版本A直接借鉴？
- 给出嫁接到版本A的具体改写示例

不要给对方优化建议。
```

### Step 4：保存报告

将完整对比报告保存为工作目录下的 `{产品名}_竞品对比报告.md` 文件。

### Step 5：询问用户

输出报告后询问：
- "需要我根据对比结果直接优化我们的文案吗？"
- "需要针对某个具体维度深入分析吗？"

如果用户要求优化，直接修改对应部分并输出对比（旧 → 新）。

## 对比原则

### 严苛性
- 打分要严苛，不允许礼貌性给高分
- 如果竞品某方面确实更能卖货，直接承认
- 评判标准是"能不能让50岁+的人掏钱"，不是"写得漂不漂亮"

### 纯文案聚焦
- 只评判文字、结构、叙事的销售杀伤力
- 价格策略、套餐组合、退款政策、支付方式、物流承诺等商业运营决策不纳入评分
- 倒计时、库存预警等运营工具不算文案分

### 人群锚定
- 所有评判都站在45-50岁+目标人群的视角
- "好文案"的标准是：这群人看了会不会掏钱
- 50岁的人信"菜市场口碑"不信"品牌杂志推荐"

### 实用性
- 优化建议只给我方（版本A），不给竞品
- 建议必须精确到具体段落、原文、改写
- "补短板+抄优点"两个角度确保可直接执行

## 多样本聚合模式（Multi-Sample Aggregation Mode）

### 何时使用

默认单样本模式下，一次 Compare 结果会受 Agent 主观判断差异 + WebFetch 抓取切片差异的影响，实测分数可能在 ±3~6 分之间漂移。当下游流水线（如 `/run-all-max`）需要用得分做阈值判定（如 "+8 分领先 = 停止迭代"）时，单样本置信度不够。此时应切换到多样本聚合模式。

### 调用方式

调用方（通常是上层 orchestrator 或用户明确要求）在 Prompt 中传入：

```
aggregation_mode = N  （N ∈ {2, 3, 5}，推荐 N=3）
```

当 `N >= 2` 时，切换到下面的流程；当 N=1 或未传，按本文件前述的单样本流程执行（向后兼容）。

### 流程

#### Step A：并行 spawn N 个独立 Compare sub-agent

- 每个 sub-agent 独立执行完整的 Step 1-3（抓取 → 打分 → 生成完整对比报告）
- N 个 sub-agent 之间**不共享**：WebFetch 缓存、打分结果、上下文
- **必须并行 spawn**（墙钟时间与单次相同，仅 token 成本变 N 倍）

##### ⚠️ 强制约束（防止递归爆炸与文件覆盖）

**约束 1：禁止递归（防护 1）**
- sub-agent 的 Prompt 中**绝对禁止**出现 `aggregation_mode` 参数
- sub-agent 必须在单样本模式运行（等同于 aggregation_mode=1）
- 如果 Orchestrator 把 aggregation_mode=N 顺手传给 sub-agent，会造成 N² 倍成本或无限递归
- Orchestrator 必须在 sub-agent Prompt 中明确写："只执行本文件 Step 1-5 的单样本流程，不要读「多样本聚合模式」章节"

**约束 2：强制覆盖默认输出路径（防护 2）**
- copy-compare 单样本默认保存路径是 Step 4 规定的 `{产品名}_竞品对比报告.md`（单数位置）
- 如果 N 个 sub-agent 都按默认路径保存，它们会互相覆盖，最终只剩 1 份
- Orchestrator **必须**在每个 sub-agent 的 Prompt 中显式指定输出路径：
  `{产品名}_对比报告_{版本}_sample{i}.md`
- 并明确写："不要按 Step 4 的默认路径保存"

**推荐的 sub-agent Prompt 模板**（Orchestrator 必须严格使用）：
```
你是直接回应文案专家（Direct Response Copywriter）。

【严格约束】
1. 读取 copy-compare/SKILL.md，但只执行 Step 1-5 的单样本流程
2. 不要读本文件末尾的「多样本聚合模式」章节（你是被 Orchestrator 调用的 sub-agent，不是 Orchestrator 本身）
3. 不要在你的工作中使用或传递 aggregation_mode 参数
4. 本次保存路径 = {产品名}_对比报告_{版本}_sample{i}.md（i 由 Orchestrator 指定）
   不要按 Step 4 默认的 {产品名}_竞品对比报告.md 路径保存

【任务】
版本 A（我方文案）：{我方文案路径}
版本 B（竞品）：{竞品URL列表}
你的 sample 编号：i = {1/2/3/.../N}

按 Step 1-5 独立打分，生成完整报告保存到上述指定路径。
```

#### Step B：主 Orchestrator 聚合 N 份结果

1. **中位数得分**
   - 我方总分 = median(N 个样本的我方总分)
   - 竞品总分 = median(N 个样本的竞品总分)
   - 领先差值 = 我方中位数 - 竞品中位数
   - 维度分项得分也各自取中位数

2. **离散度 / 置信度**
   - 我方离散度 = max(我方样本得分) - min(我方样本得分)
   - 竞品离散度 = max(竞品样本得分) - min(竞品样本得分)
   - 置信度判定规则（取竞品和我方离散度的较大者）：
     - 离散度 <= 3 分 → **高置信度**
     - 离散度 4~6 分 → **中等置信度**
     - 离散度 > 6 分 → **低置信度，建议人工复核或增加样本数**

3. **共识短板 / 共识亮点**
   - 取 N 份报告 TOP 短板/亮点的**交集**
   - 在 >= ⌈N/2⌉ 份报告里被提到的才算"共识"（N=3 时要 >=2 份提到）
   - 只出现在 1 份报告里的短板/亮点标注为"孤证，仅供参考"，不纳入主要结论

#### Step C：生成综合对比报告

**主文件**：`{产品名}_对比报告_{版本}.md`（下游流水线使用，格式向前兼容）
- 保留单样本报告的四部分结构，其中分数全部用**中位数**
- **必须**在报告顶部明文包含字符串 **`多样本打分分布`**（作为下游版本检测标记，不要省略此关键词）
- 在"第一部分：打分表"后新增一节 "第1.5部分：多样本打分分布"：
  ```
  | 样本 | 我方总分 | 竞品总分 | 差值 |
  |------|---------|---------|------|
  | Sample 1 | X | X | ±X |
  | Sample 2 | X | X | ±X |
  | Sample 3 | X | X | ±X |
  | **中位数** | **X** | **X** | **±X** |
  | 离散度 | X | X | - |
  | **置信度** | - | - | **高/中/低** |
  ```
- 在"第四部分：我方版本优化建议"前新增 "第3.5部分：共识信号 vs 孤证信号"，把 N 份报告里的短板/亮点按"共识 >= ⌈N/2⌉ 提到"和"孤证仅 1 次提到"分类展示

**附件**：N 份 sample 原始报告保留在 `{产品名}_对比报告_{版本}_sample{i}.md`，供查证

### 下游阈值使用建议

单样本模式下的 "+8 分领先 = CLEAR_WIN" 阈值，在多样本模式下细化为两个条件（**同时满足**才算 CLEAR_WIN）：

1. **主条件**：中位数领先 >= 8 分
2. **稳定性条件**：N 个样本中至少 ⌈N/2⌉ 份样本领先 >= 5 分（避免只靠一次异常高分拉起中位数）

如果主条件满足但稳定性条件不满足，标记为 "MEDIAN_WIN_UNSTABLE"，建议人工复核而非自动停止迭代。

### 成本对照

| 模式 | 墙钟时间 | Token 成本 | 置信度 |
|------|---------|-----------|--------|
| 单样本（N=1） | 1× | 1× | 低（±3~6 分漂移） |
| **多样本 N=3**（推荐） | 1×（并行） | 3× | 高 |
| 多样本 N=5 | 1×（并行） | 5× | 更高，但 ROI 递减 |

N=3 已足以覆盖大部分噪声；N=5 仅在关键决策（投放上百万预算前）或 N=3 判定为低置信度时建议追加。
